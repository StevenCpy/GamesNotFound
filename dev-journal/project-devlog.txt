What I learned today:
- To get width and height of HTML element,
1. Define a Ref: const playableAreaRef = useRef(null).
2. Assign the Ref to the element: <div ref={playableAreaRef}></div>.
3. Get width and height: const width = playableAreaRef.current.clientWidth; const height = playableAreaRef.current.clientHeight.
Note that playableAreaRef.current.clientWidth should be inside a useEffect().  This waits for the DOM to render before running the code inside useEffect().
- When using the useEffect() hook, adding
1. an empty dependency array eg. useEffect(() => { ...codeBlock }, []) makes ...codeBlock run only once (when the component is rendered for the first time).
2. no dependency array eg. useEffect(() => { ...codeBlock }) makes ...codeBlock run at every component re-render.
3. a non-empty dependency array eg. useEffect(() => { ...codeBlock }, [state]) makes ...codeBlock run every time state changes.
- Remount vs re-render: When state changes and component re-renders, its children functions are remounted (i.e. new children functions are created) which causes
any useEffect() inside the child function to re-rerun.
- UseEffect() called only when the DOM in the component is rendered.
- If I just need to update a state through a child component, I can just pass "setState" as prop or through context.
For eg. if I want the child component to increment "score", I just pass "setScore" and then call setScore(prev => prev+1) in the child component.

What I learned today:
- CSS
-- Use NavLink instead of Link to allow styling the active link.  NavLink becomes active when the current route matches its "to" prop.
- JWT TOKEN
-- On server:
--- Token can be generated by using PyJWT library.
--- Method to convert payload into token depends on encryption algorithm.  For eg. HS256 requires only a secret key, while RSA requires a private and public key.
https://pyjwt.readthedocs.io/en/stable/usage.html
--- To encode using HS256, we need a JSON payload and a secret key (that we should store in .env).  For eg. "encoded = jwt.encode(payload, key, algorithm="HS256").
--- To decode using HS256, we need the encoded token and the same secret key.  For eg. "payload = jwt.decode(encoded, key, algorithms="HS256").
-- On client
--- To send token as part of header, we add the "headers" field to our request as follows:
    headers = {
        "Authorization": `Bearer ${token}`
    }
-- Back to server:
--- On receiving a token in the header, we can access it as a param as follows:
    "Authorization: Annotated[str | None, Headers(convert_underscores=False)] = None"
    1. Annotated indicates where the data comes from.  It is of the form "Annotated[data | metadata]".
    2. convert_underscores=False is needed if the token uses "-", as fastAPI converts them to "_" by default.
    3. We can use a different param value by using an alias for eg. "another_name: Annotated[str | None, Headers(alias="Authorization", convert_underscores=False)] = None".
    4. To get the actual token, it is important to remove the "Bearer " from the header.
    5. Decode the token to get the payload.

What I learned today:
- BACKEND
-- FASTAPI
    - We can organize endpoints into different files by using fastAPI's APIRouter() as follows:
        1. Put the endpoints in separate files, each file creating a router as "router = APIRouter()".
        2. We can add a prefix eg. "/auth" when defining the router.  All endpoints in the file automatically start with this prefix.
            For eg. "/auth/signup" and "/auth/login" endpoints can then be changed to "/signup" and "/login" respectively.
        3. Define the endpoints using "@router.[methodName]" instead of "@app.[methodName]".
        4. In main.py, import the router.
        5. Say the router is routerV1.  Add its routes to the main app using @app.include_router(routerV1).
    - Tags can be used to group endpoints on the Swagger UI at "[server_address]/docs".
-- PYTHON IMPORTS
    - "from .file_name import variable_name": checks current directory for file_name.
        "from ..file_name import variable_name": checks parent directory for file_name.
    - The . and .. notation cannot be used if searched directory is the root directory.
    - "Implicit Namespace Packages": Since Python v3.3, directories are automatically treated as packages,
        which allows using the . and .. notation without defining an "__init__.py" inside the directory.
    - Having an "__init__.py" allows treating the folder as a package, which makes importing from the folder cleaner.
-- API VERSIONING
    We usually create a new API version when a change in the endpoints would cause the client to break.
    When to create a new API version:
    - Updated response format.  For eg. response's "data" field was a string, but is now an object.
    - Updated API endpoint name.  For eg. moving from "/addToLibrary/{gameID}" to "/library/{gameID}".
    When new API version is NOT needed:
    - Fixing security issues.  For eg. response exposes specific Supabase error.
        This should be fixed, otherwise clients using the old APIs can still see the leaked error message.
    - Refactoring code.  This doesn't break the client, response stays the same.
    - Switching authentication method.  For eg. "/library/{username}" endpoint receives a username to display the user's library.
        When switching to JWT tokens, the endpoint becomes "/library" and receives the token in the request header instead.
        This is more of a security issue, which should be fixed.  We don't want clients to use old APIs to bypass JWT authentication.

- Spread operation "..." can be used to destructure object into props.
    For eg. "<Component {...{prop1, prop2}} />" is the same as "<Component prop1={prop1} prop2={prop2} />".
    The outer "{}" means that it's jsx logic.  "{prop1, prop2}" is shorthand object initialization.
    "..." means turn that object into props.
- To encrypt password using bcrypt:
    - Encode password to bytes.
    - Add salt (which is basically noise).
    - Encrypt using bcrypt.hashpw(bytes, salt).
    - To compare password, first encode input password to input_bytes and use bcrypt.checkpw(input_bytes, hash).
        bcrypt extracts salt from hash, then uses it to encrypt input_bytes, then compares the result to hash.
    - Salt makes each password unique, even if they are the same.
        For eg. If 2 people sign up with password "123", their salt will be different, which make their hashes different.
        This reduces risks of attacks.
- CSS:
    - "width: fit-content" makes component width only as wide as its content.
    - We can use variables in CSS as follows:
        For eg. Define in some element, --bg-color: red
        Access it in child element using background-color: var(--bg-color)

        - Modify eslint.config.js for eslint to target "ts/tsx" files.
        - Modify package.json "scripts.build" to "tsc -b && vite build" instead of "vite build" to ensure TypeScript catches
            type errors during compilation.

        - Add "allowJS: true" to tsconfig.app.json to allow JavaScript files to be imported
            (so codebase can be incrementally converted to TS).

        - Gradually convert .jsx files to .tsx, adding type annotations and resolving TypeScript errors:
            - Add type annotations to function params and return value.
            - Create types for JSON API responses and objects.
    - Optional chaining operator "?." allows accessing a property if the left value is not null or undefined.
        For eg. user?.username returns the user's username if user is neither null not undefined.
    - Types can be defined starting with "type TypeName =".  The type name should be in PascalCase.
    - Union type operator "|" is used to indicate that a variable can be of multiple types.
        For eg. "type TypeName = number | string" means that variables of type TypeName can either be a number OR a string.
    - Intersection type operator "&" is used to indicate that a variable must be of these types.
        For eg. "type TypeName = Person | Employee" means that variables of type TypeName should be of both type Person AND type Employee.
    - It is good practice for contexts to create and provide custom useContext() hooks instead of letting the consumers explicitly
        make use of useContext().
        For eg. AuthContext.tsx provides useAuth() which wraps around useContext("AuthContext") and throws an error if
        AuthContext is null.  Then the consumer just calls useAuth().
        This prevents TypeScript language service from complaining about AuthContext being possibly null if we created the context
        with "const AuthContext = createContext(null)".

What I learned today:
DOCKER:
    - 3 main files: Dockerfile, .dockerignore and docker-compose.yaml
        1. Dockerfile - Indicates how to build the image
        2. .dockerignore - What files/directories to ignore when copying files over
        3. docker-compose.yaml - Indicates how to run the container (avoids having to type thing
                                    like "docker run --env-file .env tag_name -p 8000:8000")
    - Docker uses layer-caching mechanism.  For eg. "FROM", "WORKDIR", "COPY" and "RUN" are different layers.
        If a change is detected in one of the layers, all subsequent layers are re-built.
    - Best to start by copying dependencies for eg. requirements.txt (Python) using "COPY",
        then running command that installs the dependencies ("pip install -r requirements.txt" for example),
        then finally copying the source code.
        This way, the dependencies-installation layer is cached and is not re-triggered every time source code is modified.
    - --no-cache-dir flag when installing dependencies eg. "pip install --no-cache-dir -r requirements.txt" reduces image
        size considerably.
    - Containers are isolated from host environment.  To add environment variables to docker container runtime,
        "docker run --env-file .env tag_name"
    - Container ports isolated from host machine.
        Port mapping "-p {host_port}:{container_port}" is required, where host_port is the port of the machine
        running the container, and container_port is the port the app inside the container listens to.
        For eg. -p 7000:8000 means all traffic the host sends to port 7000 (like API requests) is routed
        to port 8000 of the container (eg. server inside container listens to API requests on port 8000).
- We can dynamically set a class on HTML using document.documentElement.classList.add("className").
    "add": adds class, "remove": removes class, "toggle": adds class if not present, else removes it.
- <label> tag makes everything between the tags clickable.
- Element can be made "invisible" by setting "opacity: 0".
- Custom toggle with icon can be achieved by:

    - Set checkbox "opacity: 0" to make it "invisible", therefore hiding it.

What I learned today:
- COOKIES:
    https://starlette.dev/responses/#set-cookie
    - Key-value based, store JWT token as value.
    - "path": indicates where cookie is reachable.  For eg. if /login endpoint (which sets cookie)
        is "/api/v3/auth/login", then cookie is valid in "/api/v3/auth".
    - "domain": Domain can set cookie only for its own domain or parent domain.
                Cookie is then available to the domain, and all its subdomains.
    - "secure": cookie only sent when request URL starts with "https".
    - "samesite"
        - "strict": first-party cookie setting (Same domain).  Cookie not sent in cross-site requests.
            Eg. accessing bank from external link will keep you logged out on the bank site
            until you manually refresh the bank site.
        - "lax": first-party cookie setting (Same domain).  Cookie sent in cross-site requests only
            for certain cases.  Accessing a site from external link will log you in
            on the destination site.
        - "None": third-party cookie setting.  Allows cross-site requests.
    - "partitioned": Prevents tracking user across sites.
        If two sites have the same ads engine that uses cookies, the cookies on both
        sites will be different.
